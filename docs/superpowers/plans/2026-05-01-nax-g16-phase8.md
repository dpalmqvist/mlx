```python
def time_case(case: Case, warmup: int = 3, iters: int = 10,
              capture_path: str | None = None) -> dict:
    run = case.build()
    for _ in range(warmup):
        mx.eval(run())
    mx.synchronize()

    if capture_path is not None:
        # Capture exactly one iteration. Synchronize before stop_capture so
        # the entire kernel is recorded — start_capture/stop_capture are GPU
        # workload boundaries, not CPU-side wall markers.
        mx.metal.start_capture(capture_path)
        mx.eval(run())
        mx.synchronize()
        mx.metal.stop_capture()

    samples = []
    for _ in range(iters):
        tic = time.perf_counter()
        out = run()
        mx.eval(out)
        samples.append((time.perf_counter() - tic) * 1e3)
    mx.synchronize()
    samples.sort()
    return {
        "kernel_label": case.kernel_label,
        "shape": case.shape,
        "median_ms": statistics.median(samples),
        "min_ms": samples[0],
        "max_ms": samples[-1],
    }
```

- [ ] **Step 2: Write the probe shell**

Create `tools/probe_naxfrag32_direct_load.py`:

```python
"""Phase 8 prototype-A feasibility probe: register-only NAXFrag32 load.

Verifies that constructing a cooperative_tensor directly from a device
pointer (no threadgroup-scratch staging) produces bit-exact equality with
the existing scratch path for the (Role::Left, transpose=false) and
(Role::Right, transpose=false) descriptors.

If this probe fails (either the kernel doesn't compile, the Metal validation
layer rejects the load, or the numerical result diverges), prototype A is
not feasible on g16 and Phase 8 must fall back to B or C.
"""

import json
import sys

import mlx.core as mx
import numpy as np


def main() -> int:
    if not mx.metal.is_nax_available():
        print(json.dumps({"skip": "NAX not available"}))
        return 0
    flavor = mx.metal.nax_arch_flavor()
    if flavor != "g16":
        print(json.dumps({"skip": f"NAX flavor is {flavor}, not g16"}))
        return 0

    # 32x32 matmul, the smallest shape that exercises NAXFrag32 end-to-end.
    # If A's load_direct path is wired in (see Task 5A) and incorrect, the
    # checksum will diverge from the reference (scratch path).
    a = mx.random.normal((32, 32)).astype(mx.float16)
    b = mx.random.normal((32, 32)).astype(mx.float16)
    c = a @ b
    mx.eval(c)

    # Reference: same op via numpy on CPU.
    a_np = np.array(a, copy=True)
    b_np = np.array(b, copy=True)
    c_ref = (a_np.astype(np.float32) @ b_np.astype(np.float32))

    err = float(np.max(np.abs(np.array(c).astype(np.float32) - c_ref)))
    rel = err / max(1e-9, float(np.max(np.abs(c_ref))))
    ok = rel < 5e-3  # fp16 tolerance for 32-element accumulate

    print(json.dumps({
        "probe": "naxfrag32_direct_load",
        "max_abs_err": err,
        "max_rel_err": rel,
        "ok": ok,
    }))
    return 0 if ok else 1


if __name__ == "__main__":
    sys.exit(main())
```

## Task 4C: (Prototype C only) Barrier-scope reduction in NAXFrag32

## Troubleshooting

### Xcode capture authorization

If `mx.metal.start_capture` fails with "Failed to start capture: Capture is not enabled":
1. The process needs `MTL_CAPTURE_ENABLED=1` in the environment. Re-run with that env var set.
2. The build may need to enable Metal API validation. Set `MTL_DEBUG_LAYER=1` in addition.
3. macOS may prompt for capture permission on first attempt — approve and retry.

If the capture starts but Xcode opens with no GPU counters, the Metal Performance HUD requires Apple Silicon and a recent macOS — confirm `mx.metal.device_info()` reports the expected M-series chip.

### Build failures after kernel edits

Most likely: missed call sites for the new scratch arg (Task 4B) or descriptor mismatch (Task 5A). Search for the function name across all kernel `.h` files:

```bash
grep -rn 'gemm_loop\|NAXFrag_t::load\|NAXFrag32::load' mlx/backend/metal/kernels/steel/
```

Cross-check that every caller passes the new arity correctly. The Phase 6 transpose audit established a precedent: load+mma descriptors MUST match — if you're touching template params on `load`, also check every `tile_matmad_nax` consumer.

### Cross-arch sanity (kPacking==2 / BaseNAXFrag / g17)

The spec requires that the fix not regress kPacking==2 paths (g17 hardware). We can't run g17 on a g16 dev machine, so the validation is *structural*: every kernel-level change must be guarded by `if constexpr (NAXFrag_::kPacking == 1)` (or live behind dispatch paths only reachable from `_g16` kernel instantiations). After committing the prototype, audit the diff:

```bash
git show HEAD -- mlx/backend/metal/kernels/steel/ | grep -nE 'kPacking|simdgroup_barrier|threadgroup_barrier|load_direct|sg_scratch' | head -40
```

For prototype A: every new code path in `nax_common.h` must be inside an existing `if constexpr (kPacking == 1)` branch (typically `NAXTile::load`'s kPacking==1 body) — `load_direct` itself is a NAXFrag32 method and can't be reached from BaseNAXFrag.
For prototype B: doubled scratch and second pointer must be gated on `(NAXFrag_::kPacking == 1)`. The kernel literals (`scratch_buf[kScratchSize]`) already use a constexpr branch — confirm the doubled size is *only* doubled in that branch.
For prototype C: the simdgroup_barrier replacement is inside NAXFrag32 methods only, so kPacking==2 is structurally unreachable. No further guard needed.

If the audit shows any unguarded change to a path BaseNAXFrag also reaches, add the guard before bench.

### Bench output regresses on a non-target case

If post-fix bench regresses an SDPA or gather case — those weren't in scope, but a kernel change in `nax_common.h` can leak across kernels because all NAX kernels share that header. Most likely cause: a `NAXFrag32` method change applied to a code path other than gemm. Mitigation: gate the change behind a constexpr or a new method name (e.g., `load_direct` rather than modifying `load`) so only the gemm dispatcher reaches it.

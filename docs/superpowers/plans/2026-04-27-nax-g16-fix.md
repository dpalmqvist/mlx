# NAX correctness on g16s (M4 Pro) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Re-enable NAX on Apple M4 Pro (`applegpu_g16s`) by introducing a 32×32 frag class (`NAXFrag32`) paired with the `(32, 32, 32)` `matmul2d_descriptor`, plumbed into `steel_gemm_fused_nax` as the first integration. Land this without regressing the g17+ NAX path or the non-NAX (SIMD-group) path.

**Architecture:**
- A new `NAXFrag32` class lives alongside `BaseNAXFrag` in `mlx/backend/metal/kernels/steel/gemm/nax.h`. It implements the same compile-time interface (`kFragRows/kFragCols/kElemsPerFrag/...`, `mma`, `load`/`store`, `row_reduce`, `row_bin_op`, etc.) but with 32×32 tiles, 32 elements/thread, single-frag MMA, and I/O routed through `metal::tensor_inline` views + `cT.load(handle)/cT.store(handle)` to avoid hard-coding the per-thread element coordinates. Note: `dextents`, `tensor`, `tensor_inline`, `execution_simdgroup` all live in `metal::` (NOT `mpp::`) on this Xcode SDK; only `matmul2d` / `matmul2d_descriptor` are in `mpp::tensor_ops`. Discovered during Task 1.4 (commit `035f04b1`).
- Kernel sources gain a g16-specific instantiation tag (`_g16`) that selects `NAXFrag32`. Existing g17+ instantiations are untouched.
- The runtime (`device.cpp`) exposes `is_nax_available()` (re-enabled for g16s) plus a new flavor predicate `nax_arch_flavor()` returning `g16` or `g17p`. Dispatchers in `matmul.cpp` (and later `quantized.cpp`, `scaled_dot_product_attention.cpp`) pick the kernel name from that flag.
- Validation is **on-hardware (M4 Pro)** at every phase boundary, using `tools/probe_nax_descriptor.py`, a new `tools/probe_nax_frag32.py`, the dense matmul test suite, and `tools/check_qmm_nax_correctness.py` (later phases).

**Tech Stack:** C++ / Metal Shading Language, Apple `MetalPerformancePrimitives` (`mpp::tensor_ops::matmul2d`, `mpp::tensor_inline`), MLX C++ backend, Python tests via `mlx.core`, CMake. Hardware: Apple M4 Pro (`applegpu_g16s`).

## Scope Discipline

This plan covers **Phase 0 (validation harness) → Phase 1 (NAXFrag32) → Phase 2 (steel_gemm_fused_nax integration)** end-to-end. After Phase 2 lands and is validated on g16s, follow-up plans will cover (in order):

- Phase 3: `steel_gemm_splitk_nax`, `steel_gemm_segmented_nax`, `steel_gemm_gather_nax` migration.
- Phase 4: `quantized_nax` and `fp_quantized_nax` migration.
- Phase 5: `steel_attention_nax` migration.
- Phase 6: Final re-enable, full validation suite, perf measurement.

The reason for the split: until Phase 2 proves that `tensor_inline` + threadgroup staging actually composes with the existing tg-memory pipeline at `(32,32,32)`, planning the details of phases 3–6 is wasted effort — concrete integration constraints will force changes to the design.

**Non-regression requirement (every phase):** The g17+ NAX path, the non-NAX SIMD-group path, and the NAX-disabled path (`MLX_METAL_NO_NAX=1`) must remain bit-identical to baseline. Validation gates explicitly test this.

def make_source_tg_staged(M, N, K, relaxed):
    rp = "true" if relaxed else "false"
    return f"""
constexpr int M={M}, N={N}, K={K};
constexpr auto desc = matmul2d_descriptor(
    M, N, K,
    /*transpose_left=*/false,
    /*transpose_right=*/false,
    /*relaxed_precision=*/{rp},
    matmul2d_descriptor::mode::multiply);

threadgroup float tgA[M*K];
threadgroup float tgB[K*N];

uint tid = thread_position_in_threadgroup.x;
for (uint i = tid; i < M*K; i += 32) tgA[i] = ((device float*)A)[i];
for (uint i = tid; i < K*N; i += 32) tgB[i] = ((device float*)B)[i];
threadgroup_barrier(mem_flags::mem_threadgroup);

matmul2d<desc, execution_simdgroup> op;
auto ctA = op.get_left_input_cooperative_tensor<float, float, float>();
auto ctB = op.get_right_input_cooperative_tensor<float, float, float>();
auto ctC = op.get_destination_cooperative_tensor<decltype(ctA), decltype(ctB), float>();

dextents<int32_t, 2> extA(M, K);
dextents<int32_t, 2> extB(K, N);
dextents<int32_t, 2> extC(M, N);
tensor<threadgroup float, dextents<int32_t, 2>, tensor_inline> tA(tgA, extA);
tensor<threadgroup float, dextents<int32_t, 2>, tensor_inline> tB(tgB, extB);
tensor<device      float, dextents<int32_t, 2>, tensor_inline> tC(C, extC);

ctA.load(tA);
ctB.load(tB);
for (uint16_t i=0; i<ctC.get_capacity(); ++i) ctC[i] = 0.0f;
op.run(ctA, ctB, ctC);
ctC.store(tC);
"""

### Task 1.2: Add `NAXFrag32` skeleton with `get_coord` and constants

**Why:** Establishes the new class. We add `get_coord` first because everything else (load/store/row_reduce) depends on it. The lane formula comes from `tools/probe_nax_descriptor.py`'s historical layout-dump output, recorded in the spec.

```cpp
///////////////////////////////////////////////////////////////////////////////
// NAXFrag32 — 32x32 frag, single-frag MMA, layout-agnostic I/O via
// tensor_inline. Used on g16 (M4 Pro), where matmul2d is only correct for
// the (32, 32, 32) descriptor. See
// docs/superpowers/specs/2026-04-27-nax-g16-fix-plan.md.
///////////////////////////////////////////////////////////////////////////////

struct NAXFrag32 {
  STEEL_CONST short kFragRows = 32;
  STEEL_CONST short kFragCols = 32;

  STEEL_CONST short kElemsPerFrag = (kFragRows * kFragCols) / 32;  // 32

  STEEL_CONST short kElemRows = 8;
  STEEL_CONST short kElemCols = 4;

  STEEL_CONST short kElemRowsJump = 1;  // unused — per-element offsets come
                                        // from dr/dc tables, not a stride.

  // One MMA produces one full 32x32 frag, i.e. unpacked.
  STEEL_CONST short kPacking = 1;

  static_assert(
      kElemRows * kElemCols == kElemsPerFrag,
      "NAXFrag32 shape is not consistent with its size");

  template <typename U>
  using dtype_frag_t = typename metal::vec<U, kElemsPerFrag>;

  // Per-thread base coordinate in the 32x32 frag. Element i lives at
  //   (a + dr[i % 8], b + dc[i / 8])
  // with
  //   a = ((lane & 1) << 1) | ((lane & 8) >> 1)        in {0, 2, 4, 6}
  //   b = ((lane & 2) >> 1) | ((lane & 4) >> 1) | ((lane & 16) >> 2)  in {0..7}
  //   dr = {0, 1, 8, 9, 16, 17, 24, 25}
  //   dc = {0, 8, 16, 24}
  // Source: tools/probe_nax_descriptor.py layout-dump variant on g16s.
  METAL_FUNC static short2 get_coord() {
    const ushort lane = __metal_get_thread_index_in_simdgroup(ushort());
    const short a = ((lane & 1) << 1) | ((lane & 8) >> 1);
    const short b = ((lane & 2) >> 1) | ((lane & 4) >> 1) | ((lane & 16) >> 2);
    return short2{b, a};  // short2 is {x = col, y = row}
  }

  METAL_FUNC static short2 dr_dc(short i) {
    constexpr short dr_table[8] = {0, 1, 8, 9, 16, 17, 24, 25};
    constexpr short dc_table[4] = {0, 8, 16, 24};
    return short2{dc_table[i / 8], dr_table[i % 8]};
  }
};
```

```cpp
  template <
      typename CType,
      typename AType,
      typename BType,
      bool transpose_a = false,
      bool transpose_b = false>
  METAL_FUNC static constexpr void mma(
      thread dtype_frag_t<CType>& C,
      const thread dtype_frag_t<AType>& A,
      metal::bool_constant<transpose_a>,
      const thread dtype_frag_t<BType>& B,
      metal::bool_constant<transpose_b>) {
    constexpr auto desc = mpp::tensor_ops::matmul2d_descriptor(
        32,
        32,
        32,
        transpose_a,
        transpose_b,
        true,
        mpp::tensor_ops::matmul2d_descriptor::mode::multiply_accumulate);

    mpp::tensor_ops::matmul2d<desc, metal::execution_simdgroup> gemm_op;

    auto ct_a =
        gemm_op
            .template get_left_input_cooperative_tensor<AType, BType, CType>();
    auto ct_b =
        gemm_op
            .template get_right_input_cooperative_tensor<AType, BType, CType>();
    auto ct_c = gemm_op.template get_destination_cooperative_tensor<
        decltype(ct_a),
        decltype(ct_b),
        CType>();

    STEEL_PRAGMA_UNROLL
    for (short i = 0; i < kElemsPerFrag; i++) {
      ct_a[i] = A[i];
      ct_b[i] = B[i];
      ct_c[i] = C[i];
    }

    gemm_op.run(ct_a, ct_b, ct_c);

    STEEL_PRAGMA_UNROLL
    for (short i = 0; i < kElemsPerFrag; i++) {
      C[i] = ct_c[i];
    }
  }
```

**Why:** The spec mandates routing I/O through `cT.load(tg_view)` so we never hard-code the `dr`/`dc` per-thread layout. This is the "layout-agnostic" path that the descriptor sweep proved correct in Task 0.1.

**Files:**

- [ ] **Step 1: Add `load` / `store` via `tensor_inline` (single-frag, no offsets)**

Inside `struct NAXFrag32`, after `mma`, add:

```cpp
  // Load the entire 32x32 frag from a contiguous threadgroup region.
  // We materialize a tensor_inline view over the threadgroup pointer and
  // use the cooperative-tensor load that the MPP runtime handles. This
  // avoids hard-coding (dr, dc) per-thread offsets — same code path as
  // tools/probe_nax_descriptor.py's tg-staged variant.
  template <typename T, typename U>
  METAL_FUNC static void load(
      thread dtype_frag_t<T>& dst,
      const threadgroup U* src,
      const short ld) {
    constexpr auto desc = mpp::tensor_ops::matmul2d_descriptor(
        32, 32, 32,
        /*transpose_left=*/false,
        /*transpose_right=*/false,
        /*relaxed_precision=*/true,
        mpp::tensor_ops::matmul2d_descriptor::mode::multiply_accumulate);
    mpp::tensor_ops::matmul2d<desc, metal::execution_simdgroup> op;
    auto ct = op.template get_left_input_cooperative_tensor<T, T, T>();

    mpp::dextents<int32_t, 2> ext(32, ld);
    mpp::tensor<threadgroup const U, mpp::dextents<int32_t, 2>, mpp::tensor_inline>
        view((threadgroup const U*)src, ext);
    ct.load(view);

    STEEL_PRAGMA_UNROLL
    for (short i = 0; i < kElemsPerFrag; i++) {
      dst[i] = static_cast<T>(ct[i]);
    }
  }

  template <typename T, typename U>
  METAL_FUNC static void store(
      const thread dtype_frag_t<T>& src,
      threadgroup U* dst,
      const short ld) {
    constexpr auto desc = mpp::tensor_ops::matmul2d_descriptor(
        32, 32, 32,
        /*transpose_left=*/false,
        /*transpose_right=*/false,
        /*relaxed_precision=*/true,
        mpp::tensor_ops::matmul2d_descriptor::mode::multiply_accumulate);
    mpp::tensor_ops::matmul2d<desc, metal::execution_simdgroup> op;
    auto ct = op.template get_destination_cooperative_tensor<
        decltype(op.template get_left_input_cooperative_tensor<T, T, T>()),
        decltype(op.template get_right_input_cooperative_tensor<T, T, T>()),
        T>();

    STEEL_PRAGMA_UNROLL
    for (short i = 0; i < kElemsPerFrag; i++) {
      ct[i] = static_cast<T>(src[i]);
    }

    mpp::dextents<int32_t, 2> ext(32, ld);
    mpp::tensor<threadgroup U, mpp::dextents<int32_t, 2>, mpp::tensor_inline>
        view((threadgroup U*)dst, ext);
    ct.store(view);
  }
```

### Task 1.7: Add `row_reduce` and `row_bin_op` for NAXFrag32

**Why:** SDPA's softmax row-reduction depends on these. Even though Phase 5 is the SDPA migration, getting the reductions right while we own the layout reasoning is much easier than retrofitting later. NAXFrag32 has 8 rows per thread (vs BaseNAXFrag's 2), so the shuffle pattern differs.

**Files:**
- Modify: `mlx/backend/metal/kernels/steel/gemm/nax.h`

- [ ] **Step 1: Derive shuffle pattern**

For `NAXFrag32`, each thread owns 8 rows × 4 cols. Threads with the same `a` (= row base) are those whose `lane & 9` are equal (i.e. lane bits 0 and 3 must match). To row-reduce horizontally across the row, we need to combine partials from threads sharing the row.

Layout formula recap: `b = ((lane & 2) >> 1) | ((lane & 4) >> 1) | ((lane & 16) >> 2)`. Threads sharing the same `a` differ only in lane bits {1, 2, 4} — that's 8 threads per row. Reduction shuffle: XOR with 2, then 4, then 16.

Add the implementation:

```cpp
  template <typename Op, typename T>
  METAL_FUNC static constexpr void row_reduce(
      thread const dtype_frag_t<T>& inp_vals,
      thread T* reduced_vals) {
    STEEL_PRAGMA_UNROLL
    for (short i = 0; i < kElemRows; i++) {
      // 4 columns per thread — combine within the thread first.
      T thr =
          Op::apply(
              Op::apply(inp_vals[i * 0 + i], inp_vals[i * 0 + i + 8]),
              Op::apply(inp_vals[i * 0 + i + 16], inp_vals[i * 0 + i + 24]));
      // Note: dtype_frag_t order is dr-major (i indexes dr), so columns for
      // the same row i are at indices i, i+8, i+16, i+24.

      // Combine across 8 threads sharing this row (lane bits 1, 2, 4 vary).
      T r = simd_shuffle_xor(thr, ushort(2));   r = Op::apply(thr, r);
      T r2 = simd_shuffle_xor(r,   ushort(4));  r2 = Op::apply(r,   r2);
      T r3 = simd_shuffle_xor(r2,  ushort(16)); r3 = Op::apply(r2,  r3);

      reduced_vals[i] = Op::apply(reduced_vals[i], r3);
    }
  }
```

- [ ] **Step 2: Add `row_bin_op`**

```cpp
  template <typename Op, typename T>
  METAL_FUNC static constexpr void row_bin_op(
      thread dtype_frag_t<T>& inp_vals,
      thread T* row_vals) {
    STEEL_PRAGMA_UNROLL
    for (short i = 0; i < kElemRows; i++) {
      // dtype_frag_t order: row i lives at indices i, i+8, i+16, i+24.
      inp_vals[i]      = Op::apply(inp_vals[i],      row_vals[i]);
      inp_vals[i + 8]  = Op::apply(inp_vals[i + 8],  row_vals[i]);
      inp_vals[i + 16] = Op::apply(inp_vals[i + 16], row_vals[i]);
      inp_vals[i + 24] = Op::apply(inp_vals[i + 24], row_vals[i]);
    }
  }
```

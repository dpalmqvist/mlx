> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restore correct output for `mx.fast.scaled_dot_product_attention` on Apple gen-16 GPUs by porting the SDPA NAX kernel to use `NAXFrag32`. Concretely: deduplicate the NAX class hierarchy across the gemm and attn trees, parameterize the SDPA NAX kernel on `class NAXFrag_`, and add `_g16` Metal instantiations using **wm=2** (so `SM = bq/wm = 32` matches NAXFrag32's 32×32 fragment).

**Architecture:** Five layers. (1) Dedup `BaseNAXFrag/NAXFrag32/NAXTile` into a shared `nax_common.h`. (2) Audit attn/ tree for tree-specific NAX usage. (3) Parameterize `attention_nax` kernel on `NAXFrag_`. (4) Add `_g16` Metal instantiations with `wm=2` (vs the regular `wm=4`). (5) Dispatcher overrides `wm=2` and appends `_g16` to kname on `kG16`. New SDPA-specific transpose probe acts as the regression gate.

**Tech Stack:** Metal Shading Language, Apple MetalPerformancePrimitives `matmul2d` + `cooperative_tensor`, MLX C++ runtime, Python test harness.

## Self-review notes (writer's notes — leave for the implementor)

- **Spec coverage**: All 6 components map to tasks. Audit (Task 1), dedup (Task 2), probe (Task 3), kernel param (Task 4), instantiation (Task 5), dispatcher (Task 6).
- **Order matters**: Dedup (Task 2) must come BEFORE kernel changes (Task 4) so the SDPA kernel can `#include "../nax_common.h"` and find NAXFrag32. The probe (Task 3) is a guard/regression rather than a strict RED-then-GREEN.
- **Risk: tree-specific code in attn/**: If Task 1's audit surfaces tree-specific code in `attn/nax.h`, Task 2 step 3 must preserve it (extension after the include). The plan currently assumes strict-subset.
- **Risk: Wm=2 perf regression**: Phase 7 changes the threadgroup size on g16 from 128 to 64 threads. This may regress non-correctness perf metrics. Out of scope for this phase; flag in PR for follow-up benchmarking.
- **Static_assert as design enforcement**: Task 4 adds `static_assert(SQ % kFragRows == 0)`. This is the same pattern from Phase 1+5 — failure-to-compile prevents accidentally instantiating an invalid combination (e.g., `wm=4 + NAXFrag32`).
- **The probe is intentionally weak**: existing `probe_naxfrag32_transpose.py` already covers all 4 (ta, tb) configs at NAXFrag32-level. The new SDPA probe adds a SDPA-flavored label so future failures are clearer about which production path is broken. Don't over-engineer it.
